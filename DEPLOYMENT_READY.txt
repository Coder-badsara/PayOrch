1. STATE MACHINE ✅
   • 12+ explicit states (CREATED → ROUTE_SELECTED → AUTH_INITIATED → ... → REFUNDED)
   • Deterministic transitions with validation
   • Pessimistic locking (SELECT FOR UPDATE)
   • Immutable audit trail with full context
   • Location: apps/payments/state_machine.py

2. INTELLIGENT ROUTING ✅
   • Multi-criteria scoring algorithm (5 factors)
   • Success Rate: 35% | Latency: 20% | Cost: 20% | Health: 15% | Fit: 10%
   • Dynamic weight configuration (database-driven)
   • Gateway health scoring
   • Real-time metrics aggregation
   • Location: apps/orchestrator/router.py

3. FAILOVER MECHANISM ✅
   • Automatic gateway switching with < 2 second timeout
   • Configurable: FAILOVER_TIMEOUT_MS, MAX_FAILOVER_ATTEMPTS
   • Concurrent request execution with ThreadPoolExecutor
   • Fallback to alternate gateways on failure
   • Complete routing history tracking
   • Location: apps/orchestrator/failover.py

4. CIRCUIT BREAKER ✅
   • Three states: CLOSED → OPEN → HALF_OPEN
   • Automatic trip on 70% failure rate
   • Recovery timeout: 60 seconds (configurable)
   • Per-gateway and per-payment-method tracking
   • Real-time health monitoring
   • Location: apps/orchestrator/health_service.py

5. IDEMPOTENCY LAYER ✅
   • Client-generated UUID keys
   • Database-level deduplication
   • PostgreSQL advisory locks support (when on Postgres)
   • Request hash verification
   • TTL-based cleanup (86400 seconds default)
   • Location: apps/idempotency/service.py

6. WEBHOOK PIPELINE ✅
   • Signature verification (Razorpay, Stripe, PayU, UPI)
   • HMAC-SHA256, HMAC-SHA512 support
   • Atomic deduplication with unique constraints
   • Persistent queue with retry logic
   • Dead Letter Queue (DLQ) for failed webhooks
   • Out-of-order delivery handling
   • Celery-based async processing
   • Location: apps/webhooks/service.py

7. HEALTH MONITORING ✅
   • Per-minute sliding window (100-event buffer)
   • Success rate calculation
   • P95 latency tracking
   • Error rate monitoring
   • Real-time status updates
   • Location: apps/orchestrator/health_service.py

8. RECONCILIATION ENGINE ✅
   • Detects stale transactions (> 5 minutes)
   • Compares internal records vs gateway source of truth
   • Records discrepancies in ReconciliationLog
   • Periodic task with Celery support
   • Manual trigger via: python manage.py run_reconciliation
   • Location: apps/orchestrator/tasks.py

9. AUDIT TRAIL ✅

⚠️ IMPORTANT NOTES

1. PRODUCTION DEPLOYMENT:
   • Change DEBUG=False in backend/.env
   • Use PostgreSQL instead of SQLite
   • Set up Gunicorn/uWSGI + Nginx reverse proxy
   • Use environment-specific .env files
   • Enable HTTPS/SSL
   • Use Docker for containerization

2. SECURITY:
   • Backend .env contains demo API keys
   • Use proper secrets management in production
   • Implement rate limiting (django-ratelimit already installed)
   • Add authentication/authorization layer as needed

3. DATABASE MIGRATION TO POSTGRESQL:
   Set DATABASE_URL in .env:
   DATABASE_URL=postgresql://user:pass@host:5432/db_name
   
   Then:
   python manage.py migrate

4. CELERY CONFIGURATION:
   For production, configure:
   • CELERY_BROKER_URL (message queue)
   • CELERY_RESULT_BACKEND (result storage)
   • Celery workers (separate processes)

❓ TROUBLESHOOTING

If backend won't start:
  → Check: python manage.py check
  → Reinstall: pip install -r requirements.txt
  → Clear: rm backend/db.sqlite3 && python manage.py migrate

If frontend won't load:
  → Check: http://localhost:8000/api/v1/health/
  → Reinstall: npm install --legacy-peer-deps
  → Clear cache: rm -rf frontend/node_modules && npm install

If ports are in use:
  → Backend: python manage.py runserver 0.0.0.0:8001
  → Frontend: Vite auto-increments (5174, 5175, etc.)

If CORS errors occur:
  → Check CORS_ORIGINS in backend/.env
  → Ensure frontend URL is listed
